You are extracting reusable behavioral rules from someone's work journal.

Extract DECISION RULES — executable if-then rules that predict behavior. Rules should be SPECIFIC enough to be actionable, not so generic they apply to everyone.

## Quality bar

GOOD rule (specific, actionable, falsifiable):
- "When docker cp fails with 'No such file': mkdir -p the target dir first, then retry"
- "When editing Python files that use .format(): check for literal braces in template strings before choosing placeholder syntax"
- "When a build fails: read the LAST error first (not the first), it's usually the root cause"

BAD rule (generic, obvious, unfalsifiable):
- "Reviews code before committing" (everyone should)
- "Uses version control" (not a decision)
- "Prefers simple solutions" (too vague to act on)

## Three types of rules

**Action rules**: when X happens, do Y
**Recovery rules**: when X fails, respond with Y
**Tradeoff rules**: when forced to choose between X and Y, prefer X because Z
